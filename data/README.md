    Аналіз прогнозних показників розвитку України (2026-2028)

Метою цього проекту є аналіз, дослідження, прогнозування та візуалізація основних прогнозних макропоказників економічного і соціального розвитку України на період 2026-2028 роки. Проєкт спрямований на аналіз ключових економічних індикаторів, схвалених Кабінетом Міністрів України, для розуміння очікуваної динаміки ВВП, ринку праці та зовнішньої торгівлі.

8. Контейнеризація та Docker Workspace (Лабораторна робота №3)

У межах цієї частини проєкт було повністю переведено на мікросервісну архітектуру. 
Додаток розділено на 5 ізольованих сервісів, які оркеструються за допомогою Docker Compose, 
взаємодіють через спільні томи (Volumes) та релевантно діляться станом за допомогою СУБД SQLite.

### 1. Структура проєкту (Docker Workspace)
```text

│   ├── data_quality_analysis.py  # Сервіс аналізу якості
│   ├── data_research.py          # Сервіс розрахунку відповідей на 3 питання

│   ├── web.py                    # Веб-дашборд користувача (Streamlit)
```

### 2. Опис мікросервісів додатка
data_load: Точка входу додатка. Зчитує файл dataset.xls за допомогою двигуна xlrd і розгортає реляційну базу даних SQLite, 
автоматично створюючи таблицю macro_table.
quality_analysis: Сервіс аналізу цілісності даних. Робить вибірку з SQLite, перевіряє датасет на наявність пропусків (NaN), 
дублікатів та аналізує типи даних колонок. Генерує звіт quality_report.txt.
research: Модуль аналітичних маркерів. Проводить контекстний пошук за ключовими словами для вирішення трьох дослідницьких 
питань (прогноз торгівлі, динаміка безробіття, пік ВВП). Експортує результати у структурований research_report.json.
visualization: Сервіс графічної звітності. Будує два графіки: порівняльну гістограму експорту/імпорту України та лінійний 
тренд зниження рівня безробіття (2026-2028). Зберігає зображення economic_analysis.png.
web: Серверний веб-інтерфейс на базі Streamlit. Він працює як клієнтська панель (дашборд), що зчитує згенеровані іншими 
модулями артефакти з тому й презентує графіки та результати розрахунків користувачу через браузер.

### 3. Організація обміну даними (Docker Volumes)
Взаємодія між контейнерами побудована без використання прямого мережевого обміну, а за допомогою механізмів спільного 
доступу до файлової системи та суворої оркестрації запуску:
shared_db — використовується для шерингу файлу бази даних database.db, створеного сервісом data_load. Інші аналітичні 
модулі мають до нього доступ у режимі читання.
shared_reports — виступає спільним сховищем текстових, JSON звітів та графічних фігур. Аналітичні сервіси записують туди 
артефакти, а сервіс web зчитує їх для відображення.
Щоб уникнути помилок доступу до даних (Race Conditions), у файлі конфігурації налаштовано послідовність виконання за допомогою 
умови service_completed_successfully.

Хмарне розгортання: Terraform + cloud-init (Лабораторна робота №4)
У межах цієї роботи реалізова но повністю хмарний сценарій деплою інфраструктури в Microsoft Azure без використання локального 
комп'ютера розробника. Створення ресурсів повністю автоматизовано засобами Terraform, а конфігурація операційної системи Linux 
VM та запуск Docker-додатка виконуються під час першого старту за допомогою технології cloud-init.

1. Структура IaC каталогів проєкту
open-data-ai-analytics/
├── data/
│   └── raw/
│       └── dataset.xls       # Вхідний макроекономічний датасет Excel
├── src/
│   ├── data_load.py, quality_analysis.py, data_research.py, visualization.py, web.py
│   ├── Dockerfile            # Уніфікований збірник образів
│   └── requirements.txt      # Залежності для контейнерів
├── infra/
│   └── terraform/            # Каталог конфігурації інфраструктури як код (IaC)
│       ├── main.tf           # Головний файл маніфесту ресурсів Azure
│       ├── variables.tf      # Конфігурація вхідних змінних
│       ├── outputs.tf        # Сценарій виведення публічного IP
│       └── cloud-init.yaml   # Скрипт автоматичного налаштування Linux VM
└── compose.yaml              # Маніфест мультиконтейнерної оркестрації Docker

2. Опис створених хмарних ресурсів Azure
azurerm_resource_group: Логічна група ресурсів для ізоляції компонентів проєкту.
azurerm_virtual_network & azurerm_subnet: Віртуальна приватна мережа та підмережа для внутрішньої комунікації компонентів.
azurerm_public_ip: Динамічна публічна IP-адреса для доступу до веб-панелі з глобальної мережі Інтернет.
azurerm_network_security_group: Брандмауер мережі з налаштованими правилами вхідного трафіку: порт 22 (SSH) та порт 8501 (Streamlit-сервер).
azurerm_linux_virtual_machine: Хмарний Linux-сервер на базі Ubuntu 22.04 LTS (розмір Standard_B1s) для розгортання Docker Workspace.

3. Алгоритм роботи cloud-init сценарію
Під час першого завантаження (first boot) створеної віртуальної машини, утиліта cloud-init автоматично виконує такі інфраструктурні кроки:
Оновлює системні пакети менеджера apt.
Встановлює офіційний Docker Engine та плагін оркестрації docker-compose-plugin.
Активує та запускає демон Docker у фоновому режимі ОС.
Клонує цей актуальний репозиторій з GitHub у робочу директорію сервера.
Запускає команду "docker compose up --build -d" для розгортання та старту всіх 5 мікросервісів додатка.
Інструкція із запуску проєкту в хмарі
Крок 1. Відкриття Azure Cloud Shell
Увійдіть у свій акаунт на Azure Portal (https://portal.azure.com). У верхньому правому кутку натисніть на іконку Cloud Shell 
(значок терміналу). Переконайся, що в налаштуваннях середовища обрано Linux Bash.
Крок 2. Клонування репозиторію та генерація ключів
У вікні Cloud Shell виконайте команди для завантаження коду IaC:
git clone https://github.com/yur1k14/open-data-ai-analytics.git
cd open-data-ai-analytics/infra/terraform
Якщо в хмарному оточенні Cloud Shell ще немає налаштованих ключів безпеки, згенеруйте SSH-ключ для авторизації на майбутній 
віртуальній машині (на всі запити терміналу натискайте клавішу Enter):
ssh-keygen -t rsa -b 4096 -f ~/.ssh/id_rsa -N ""
Крок 3. Створення інфраструктури за допомогою Terraform
Послідовно виконайте обов'язковий набір IaC команд у консолі:
terraform init
terraform fmt
terraform validate
terraform plan
terraform apply -auto-approve
Дочекайтеся завершення процесу створення (близько 1-2 хвилин). Наприкінці у блоці "Outputs" з'явиться сформована публічна 
IP-адреса віртуальної машини, наприклад: public_ip_address = "40.121.32.145".
Крок 4. Перевірка результатів розгортання
Після завершення роботи Terraform, хмарній системі cloud-init потрібно близько 2 хвилин для первинного налаштування Docker 
та клонування коду на сервері.
Перевірити працездатність та HTTP-відповідь Streamlit-сервера можна прямо з Cloud Shell за допомогою утиліти curl (замініть 
IP на адресу з вашого виводу):
curl http://40.121.32.145:8501
Для перегляду повноцінного графічного інтерфійсу додатка, відкрийте нову вкладку браузера та перейдіть за адресою: http://ВАШ_PUBLIC_IP:8501
Крок 5. Видалення хмарних ресурсів
Після завершення демонстрації та захисту лабораторної роботи обов'язково знищте створену інфраструктуру в Azure Cloud Shell

Моніторинг інфраструктури та сервісів: Prometheus + Grafana (Лабораторна робота №5)
У межах цієї лабораторної роботи розгорнуту в Microsoft Azure систему було доповнено повноцінним екосистемним стеком моніторингу 
та спостережуваності (Observability). Конвеєр збирає, агрегує та візуалізує часові ряди метрик із трьох рівнів архітектури: 
апаратного стану віртуальної машини, показників життєдіяльності Docker-контейнерів та прикладних метрик самого веб-застосунку 
Streamlit. Керування розгортанням та налаштуванням так само повністю виконується всередині хмарного середовища Azure Cloud Shell 
без використання локальних потужностей.

1. Доповнена структура каталогів проєкту
open-data-ai-analytics/
├── src/                      # Аналітичні модулі та дашборд Streamlit
├── infra/
│   └── terraform/            # Маніфести main.tf, variables.tf, outputs.tf з оновленими NSG правилами
├── monitoring/               # Новий виділений каталог систем спостережуваності
│   ├── prometheus/
│   │   └── prometheus.yml    # Конфігурація цілей та інтервалів збору метрик
│   └── docker-compose.monitoring.yml # Маніфест оркестрації сервісів моніторингу
└── compose.yaml              # Основний compose файл аналітичного застосунку

2. Опис розгорнутих сервісів моніторингу
prometheus: Ядро системи моніторингу. Здійснює збір метрик методом періодичного опитування (scrape) налаштованих цілей, зберігає 
часові ряди в інтегрованій TSDB базі даних та надає інтерфейс для виконання PromQL запитів.
grafana: Платформа візуалізації та аналітики. Підключається до Prometheus як до джерела даних (Data Source) без додаткових 
плагінів та відображає динамічні графіки стану системи на інтерактивних панелях.
node-exporter: Агент збору апаратних метрик Linux. Монтується безпосередньо в системні директорії хоста (proc, sys, rootfs) 
та експортує дані про утилізацію процесора, пам'яті, мережевого трафіку та дисків віртуальної машини.
cadvisor: Аналітик контейнеризації від Google. Автоматично ізолює, агрегує та експортує метрики продуктивності (CPU, RAM, 
ліміти, мережа) для кожного запущеного в Docker-просторі контейнера.

3. Налаштовані цілі збору даних (Scrape Targets)
У конфігураційному файлі prometheus.yml налаштовано чотири завдання для безперервного моніторингу:
job_name: prometheus — збір внутрішніх метрик самої системи моніторингу (порт 9090).
job_name: node-exporter — отримання телеметрії завантаження ядра Linux VM хоста (порт 9100).
job_name: cadvisor — моніторинг ресурсів мікросервісів додатка (порт 8080).
job_name: streamlit-app — збір прикладних метрик веб-інтерфейсу (порт 8501, ендпоінт /metrics).
Взаємодія між контейнерами аналітики та моніторингу реалізована за допомогою підключення сервісів Prometheus, Grafana, ]
Node Exporter та cAdvisor до спільної зовнішньої мережі основного застосунку open-data-ai-analytics_default.

5. Інструкція із запуску та перевірки моніторингу в хмарі
Крок 1. Оновлення інфраструктури. Увійдіть в Azure Cloud Shell, перейдіть у каталог проєкту та підтягніть зміни з GitHub: 
cd ~/open-data-ai-analytics && git pull origin main. Перейдіть у папку IaC та застосуйте оновлений план для відкриття портів 
3000 і 9090 у брандмауері Azure: cd infra/terraform && terraform apply -auto-approve.
Крок 2. Валідація збору метрик. Скопіюйте публічну IP-адресу з виводу Terraform. Відкрийте у браузері сторінку Prometheus Targets 
за адресою http://PUBLIC_IP:9090/targets і переконайтеся, що всі чотири цілі мають статус UP.
Крок 3. Налаштування Grafana. Відкрийте веб-інтерфейс Grafana за адресою http://PUBLIC_IP:3000. Перейдіть у меню Connections -> Data Sources, 
натисніть Add data source та оберіть Prometheus. У полі URL введіть внутрішню адресу контейнера: http://prometheus:9090. Натисніть кнопку 
Save & Test наприкінці сторінки.
Крок 4. Імпорт панелей спостереження. Створіть новий дашборд із назвою Macroeconomic Analytics Monitoring. Додайте чотири панелі, 
прописавши відповідні PromQL запити у полі Metrics browser:
Завантаження процесора VM: 100 - (avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)
Споживання оперативної пам'яті хоста: node_memory_MemTotal_bytes - node_memory_MemFree_bytes - node_memory_Buffers_bytes - 
node_memory_Cached_bytes
Загальна кількість активних контейнерів: count(container_last_seen{container_label_com_docker_compose_service!=""})
Пам'ять контейнера Streamlit: container_memory_usage_bytes{name="open-data-ai-analytics-web-1"}
Крок 5. Зупинка та очищення хмари. Після завершення фіксації результатів та демонстрації моніторингу викладачу, виконайте 
команду руйнування інфраструктури в Azure Cloud Shell, щоб повністю видалити сервер, мережі й томи та припинити тарифікацію 
студентського гранту: terraform destroy -auto-approve.

Автоматизація розгортання: Декларативний GitOps та Argo CD (Лабораторна робота №6)
У межах цієї лабораторної роботи архітектуру розгортання проєкту було повністю переведено на методологію GitOps. 
Керування станом системи виконується декларативно, де єдиним джерелом істини (Source of Truth) виступає Git-репозиторій, 
а синхронізація реального стану середовища з бажаним конфігом автоматично забезпечується безперервним опитуванням агента 
Argo CD всередині оркестратора Kubernetes.

1. Структура IaC та GitOps каталогів проєкту
open-data-ai-analytics/
├── data/
│   └── raw/
│       └── dataset.xls       # Вхідний макроекономічний датасет Excel
├── src/
│   ├── data_load.py, quality_analysis.py, data_research.py, visualization.py, web.py
│   ├── Dockerfile            # Уніфікований збірник образів
│   └── requirements.txt      # Залежності для контейнерів
├── infra/
│   └── terraform/            # Каталог конфігурації інфраструктури як код (IaC)
│       ├── main.tf           # Головний файл маніфесту ресурсів Azure
│       ├── variables.tf      # Конфігурація вхідних змінних
│       ├── outputs.tf        # Сценарій виведення публічного IP
│       └── cloud-init.yaml   # Скрипт автоматичного налаштування Linux VM
├── monitoring/               # Каталог конфігурацій систем спостережуваності (Prometheus/Grafana)
├── gitops/                   # Нова декларативна зона GitOps для Kubernetes
│   ├── app/
│   │   ├── namespace.yaml    # Створення ізольованого простору імен
│   │   ├── deployment.yaml   # Опис специфікації подів та реплік застосунку
│   │   └── service.yaml      # Мережева служба доступу NodePort
│   └── argocd/
│       └── application.yaml  # Специфікація автоматизації Argo CD Application
└── compose.yaml              # Маніфест мультиконтейнерної оркестрації Docker

2. Опис компонентів GitOps інфраструктури
k3s: Полегшений і оптимізований дистрибутив Kubernetes, розгорнутий на Azure Linux VM. Забезпечує роботу подів, сервісів 
та мережевих політик нашого застосунку.
Argo CD: GitOps-контролер, який ізольовано працює у просторі імен argocd, відстежує стан віддаленого GitHub-репозиторію 
та автоматично усуває дрейф конфігурації (Configuration Drift) без ручного застосування команд kubectl apply.
Deployment і Service маніфести: Декларативні файли, які описують запуск реплік Streamlit веб-інтерфейсу, прокидання внутрішніх 
портів (8501) та запуск зовнішнього балансувальника NodePort (30501).

3. Налаштовані порти для доступу
Сервіс web (Streamlit в Kubernetes): зовнішній порт 30501. Доступний за адресою: http://PUBLIC_IP:30501.
Сервіс Argo CD Server: зовнішній порт 30007. Доступний за адресою: https://PUBLIC_IP:30007.
Інструкція із запуску проєкту в хмарі
Крок 1. Відкриття Azure Cloud Shell
Увійдіть у свій акаунт на Azure Portal (https://portal.azure.com). У верхньому правому кутку натисніть на іконку Cloud Shell 
(значок терміналу). Переконайся, що в налаштуваннях середовища обрано Linux Bash.
Крок 2. Клонування репозиторію та генерація ключів
У вікні Cloud Shell виконайте команди для завантаження коду IaC:
git clone https://github.com/yur1k14/open-data-ai-analytics.git
cd open-data-ai-analytics/infra/terraform
Якщо в хмарному оточенні Cloud Shell ще немає налаштованих ключів безпеки, згенеруйте SSH-ключ для авторизації на майбутній 
віртуальній машині (на всі запити терміналу натискайте клавішу Enter):
ssh-keygen -t rsa -b 4096 -f ~/.ssh/id_rsa -N ""
Крок 3. Створення інфраструктури за допомогою Terraform
Послідовно виконайте обов'язковий набір IaC команд у консолі (попередньо виконайте az login для авторизації сесії провайдера):
terraform init
terraform fmt
terraform validate
terraform plan
terraform apply -auto-approve
Дочекайтеся завершення процесу створення (близько 1-2 хвилин). Наприкінці у блоці "Outputs" з'явиться сформована публічна 
IP-адреса віртуальної машини, наприклад: public_ip_address = "40.121.32.145".
Крок 4. Налаштування кластера k3s та Argo CD через SSH
Підключіться до створеної віртуальної машини з вікна Cloud Shell за допомогою SSH:
ssh azureuser@ВАШ_PUBLIC_IP
Встановіть полегшений кластер Kubernetes за допомогою офіційного автоматичного скрипта:
curl -sfL https://get.k3s.io | sh -
Перевірте успішність запуску та працездатність вузла кластера:
sudo kubectl get nodes
Розгорніть GitOps-контролер Argo CD у кластері, створивши для нього окремий простір імен:
sudo kubectl create namespace argocd
sudo kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
Переведіть службу сервера Argo CD у режим NodePort для зовнішнього доступу через браузер:
sudo kubectl patch svc argocd-server -n argocd -p '{"spec": {"type": "NodePort", "ports": [{"port": 80, "targetPort": 8080, "nodePort": 30007}]}}'
Зчитайте та скопіюйте автоматично згенерований таємний початковий пароль адміністратора для входу:
sudo kubectl -n argocd get secret argocd-initial-admin-secret -o jsonpath="{.data.password}" | base64 --decode; echo
Крок 5. Підключення автоматизації в Argo CD
Відкрийте нову вкладку свого браузера та перейдіть до веб-панелі Argo CD за адресою https://ВАШ_PUBLIC_IP:30007. 
Авторизуйтеся під користувачем admin, вставивши скопійований пароль.
Поверніться в SSH-термінал віртуальної машини та застосуйте декларативний маніфест програми, який зв'яже ваш Kubernetes з репозиторієм на GitHub:
sudo kubectl apply -f /home/azureuser/app/open-data-ai-analytics/gitops/argocd/application.yaml
В інтерфейсі браузера з'явиться картка програми, яка автоматично синхронізує поди. Після переходу в стан Synced, твій Streamlit-додаток 
стане доступним за адресою http://ВАШ_PUBLIC_IP:30501.
Крок 6. Тестування автоматичного оновлення та Rollback
Для перевірки роботи автоматизації без ручного застосування команд kubectl, відредагуйте файл gitops/app/deployment.yaml на локальному комп'ютері, 
змінивши параметр replicas з 2 на 3, та відправте зміни на GitHub (git push). Argo CD зафіксує зміни та самостійно розгорне третій под у кластері.
Для проведення операції відкату (Rollback), виконайте скасування останнього коміту у вашому локальному терміналі за допомогою команди 
git revert HEAD --no-edit та відправте зміни в репозиторій. Контролер автоматично зафіксує оновлення гілки та видалить зайву 
репліку подів, повернувши систему до початкового стабільного стану.
Крок 7. Видалення хмарних ресурсів
Після завершення демонстрації та захисту лабораторної роботи обов'язково знищте створену інфраструктуру в Azure Cloud Shell 
для збереження ліміту вашого студентського грошового кредиту:
terraform destroy -auto-approve
